What I accomplished since the previous report

This week, in collaboration with my team, I completed the core data acquisition and initial quality assessment for our zebrafish RNA-seq project (BioProject PRJNA1277581; GEO Series GSE299935). We downloaded all 30 paired-end SRRs (SRR34002410–SRR34002439) using SRA Toolkit, assessed read quality with FastQC, and began trimming with FASTX in preparation for re-running QC on the cleaned reads.

Also, in consultation with my team and following their approach, I created a simple, resumable pipeline for the team (after manual validation of the commands and first SRR data) to automate “download → QC → (trim)” efficiently without overloading shared resources. As a result, we produced outputs for all 30 SRRs (SRR34002410–SRR34002439) as paired *.fastq.gz in /home/zebrafish/sra_runs and 60 FastQC reports (ZIP/HTML per mate) in /home/zebrafish/fastqc_out, along with QC tables and figures for interpretation.

To summarize FastQC across the dataset, I used Python and pandas to parse each FastQC ZIP and calculate a per-report severity score (3×FAIL + WARN). This let us quickly see which reports (and which SRRs) concentrate the most WARN/FAIL modules, and which modules are consistently problematic across the project. In collaboration with my team, I plotted this data using Python to generate the figures below, which show the top problematic reports and the module-level summary across all 60 FastQC outputs.

Methods used (commands + parameters)

We mainly used three tools: the SRA Toolkit (fastq-dump / fasterq-dump), FastQC, and FASTX, and then used the following workflow: download → FastQC → summarize findings at the SRR + module level → (FASTX trim) → re-run FastQC → summarize findings at the SRR + module level. However, due to time constraints we were not able to analyze the trimmed data.

Manual Commands for Each SRR

fastq-dump --split-files --gzip SRR34002437 # download paired FASTQs
fastqc -t 1 SRR34002437_1.fastq.gz SRR34002437_2.fastq.gz # QC both mates
fastq_quality_trimmer -Q33 -t 20 -l 30 # FASTX: trim low-Q 3' tails (stdin->stdout)

Small Pipeline to Automate Manual Steps

The manual steps (defined above) were wrapped into a resumable pipeline with two workers (download + FastQC). The pipeline was run three times, once per member run list, so the team could run the process consistently and monitor it with simple logs. Below is an example of the pipeline execution command for me: 

PIPE=/home/pzg8794/zebrafish/scripts/sra_runs_pipeline_sra3.sh # pipeline wrapper
RUNS_FILE=/home/zebrafish/split_run_ids/runs.member.piter.txt # Piter SRRs
PIPE_DIR=/home/pzg8794/sra_runs_pipeline_piter # per-member state/logs
ACC=PRJNA1277581 MEMBER=piter RUNS_FILE="$RUNS_FILE" PIPE_DIR="$PIPE_DIR" DUMP_THREADS=1 FASTQC_THREADS=1 \
  "$PIPE" start # start download + FastQC

Analysis Methodology

In collaboration with my team, I used Python and pandas to parse FastQC output from each *_fastqc.zip, collapsed mates to one SRR-level status per module (worst-of-mates), and ranked runs by severity (3×FAIL + WARN) to prioritize cleanup. Then, I plotted the top problematic reports and the module-level summary across all 60 FastQC outputs to identify global patterns (e.g., duplication in RNA-seq) versus issues that may be improved by trimming/clipping (e.g., adapter signal). Finally, I generated an overlay plot of per-sequence GC content across all reports to check for systematic shifts or secondary peaks that could suggest contamination or a markedly different composition.

Problems encountered

First, the SRRs are large, and as instructed we downloaded one run at a time to avoid stressing shared resources, so total wall-clock time was longer than expected. Second, the server had multiple maintenance shutdowns that interrupted downloads mid-run and required resuming after restart. Lastly, on the analysis side, FastQC raised issues we need to investigate and clean (e.g., duplication flags across all SRRs), and confirming the exact paper/publication tied to this dataset is still in progress even though the dataset accessions (PRJNA1277581 / GSE299935) are known.
